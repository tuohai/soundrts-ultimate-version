; can_capture: 1 = default order on capture_hp_threshold 100 targets is occupy (capture), 0 = attack/move instead
; requirements: objects (units, upgrades) required by the object to be acquired (unit or building) or used (ability)

; ===== 时代（phase）系统示例 =====
; phase 是一种特殊的 upgrade。
; 默认情况下研究后会对玩家所有现有及未来的单位生效；
; 若设置了 phase_targets，则仅作用于匹配的单位。
;
; 关键字段：
;   phase bonus stat1 val1 stat2 val2 ...
;       两个词的关键字（与 effect bonus 对齐）；cost 等列表型属性可写多个值，
;       例如  phase bonus cost -5 -3 -2 mdg 1
;   phase_targets <type1> <type2> ...
;       可选；当设置时，phase bonus 中的非成本项仅作用于匹配的单位。
;       匹配规则：精确 type_name / is_a 继承链 / 类别（worker/soldier/building 等）
;       任一命中。前缀 ``-`` 表示排除（如 ``phase_targets -building`` 表示除建筑外
;       的所有单位）；可与正向项混用（如 ``phase_targets soldier -footman``）。
;       注意：cost/time_cost/population_cost 等成本类加成始终走玩家
;       全局，不受 phase_targets 限制。仅当 phase_targets 含正向匹配项
;       （非 ``-`` 排除）且同时写了成本类 bonus 时，引擎会发出一次性警告。
;   units_auto_upgrade 0/1
;       1 时研究完成后，仅"与本时代绑定"的形态升级会被瞬时执行：
;         单位的 can_upgrade_to 中，若存在某个目标的 requirements 列表里
;         显式包含本时代名（例如 castle_age），那个目标会被免费瞬时升级。
;       未把本时代写入 requirements 的常规手动升级链（如
;       townhall→keep→castle、scouttower→guardtower 等）不会被影响。
;       想让某形态随某时代自动取代旧形态，把该时代名加到目标形态的
;       requirements 里即可（示例：让 swordsman 在城堡时代取代 footman，
;       在 swordsman 的 requirements 里追加 castle_age）。
;       若同时设置了 phase_targets，仍先按 phase_targets 过滤源单位。
;   can_upgrade_to <next_phase>
;       仅作 UI 展示用途；链式升级请通过 requirements 实现。
;
; 在哪个建筑/单位里推进时代：把时代名加到该建筑/单位的 can_advance 列表即可。
; 默认示例把它放在 townhall（及其派生的 keep/castle）上，符合 AoE 玩家直觉，
; 但引擎不强制——任何 can_advance 列表里包含该时代名的单位都能推进时代。
; 注意：时代升级走 can_advance / advance 命令，与 can_research / research
; 命令（科技研究）相互独立；把 phase 放到 can_research 里不会再生效。

def feudal_age
